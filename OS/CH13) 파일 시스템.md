# 파일 시스템

## 왜 파일 시스템이 필요한가

파일을 저장하는 가장 단순한 방법은 디스크에 순서대로 쭉 이어 쓰는 것이다.

```
[hello.txt 내용][image.png 내용][notes.txt 내용]...
```

이 방식으로는 hello.txt를 읽을 수 없다. 어디서 시작하고 어디서 끝나는지, 이름이 무엇인지, 누가 소유자인지, 접근 권한은 무엇인지 적힌 곳이 없기 때문이다.

파일이 삭제될 때도 문제가 생긴다. 중간에 빈 공간이 생겨도 어디가 비어있는지 알 수 없고, 가변 크기의 빈 공간에 다른 파일을 끼워 넣다 보면 외부 단편화가 발생한다.

파일 시스템은 이 문제들을 해결하는 계층이다.

- 파일마다 이름, 크기, 권한, 소유자, 위치 정보를 별도로 저장한다
- 어떤 블록이 비어있는지 추적한다
- 파일 내용이 디스크 어디에 있는지 매핑한다

## inode: 파일의 신원증명서

파일 시스템은 파일 하나당 하나의 자료구조를 배정한다. inode(Index Node)다.

inode에는 파일 데이터를 제외한 모든 정보가 들어있다.

| 항목 | 내용 |
|------|------|
| 파일 타입 | 일반 파일, 디렉토리, 심볼릭 링크 |
| 파일 크기 | 바이트 단위 |
| 권한 | rwxr-xr-x 형식 (소유자/그룹/나머지 각 3비트) |
| 소유자 | UID, GID |
| 타임스탬프 | 생성, 수정, 마지막 접근 시간 |
| 데이터 블록 포인터 | 실제 내용이 저장된 디스크 블록 번호 목록 |

주목할 점이 있다. 파일 이름이 없다.

inode는 자신의 이름을 모른다. 파일 이름은 디렉토리가 관리한다. 디렉토리의 실체는 "이름 → inode 번호" 매핑 테이블이다.

```
/home/alice/ 디렉토리:
┌──────────────┬───────────┐
│  파일 이름   │ inode 번호 │
├──────────────┼───────────┤
│ "hello.txt"  │    42     │
│ "image.png"  │    77     │
│ "notes/"     │    13     │
└──────────────┴───────────┘

inode #42:
  크기: 3KB
  권한: rw-r--r--
  소유자: alice
  데이터 블록: [7, 23, 101]
```

파일을 열 때 OS는 디렉토리에서 이름을 찾아 inode 번호를 얻고, 그 inode에서 실제 데이터 블록 위치를 얻는다. 이름은 입구고, inode는 신원증명서이며, 데이터 블록은 실체다.

inode는 디스크의 inode 테이블 영역에 저장된다. 자주 접근하는 inode는 RAM에 캐싱된다(inode cache). 가상 메모리에서 TLB가 페이지 테이블 조회를 캐싱하는 것과 같은 구조다.

## 파일 열기 흐름

### 트리거: `open()` 시스템 콜

`open("/home/alice/hello.txt")` 를 호출하는 순간 커널이 경로 문자열을 받아 왼쪽부터 파싱한다. 첫 글자가 `/` 이면 절대경로다. 커널은 루트에서 시작한다.

그런데 "루트 = inode #2" 라는 사실은 어디서 왔을까.

### 슈퍼블록 — 파일 시스템의 설명서

파일 시스템을 포맷할 때 디스크 맨 앞에 **슈퍼블록**을 쓴다. 이 파일 시스템 전체에 대한 메타데이터가 담겨 있다.

```
[슈퍼블록]
블록 크기:         4KB
전체 블록 수:      1,048,576
inode 총 개수:     262,144
루트 inode 번호:   2          ← 루트(/)의 위치
inode 테이블 위치: 블록 #8부터
```

OS가 부팅할 때 루트 파일 시스템을 **마운트**한다. 마운트 과정에서 슈퍼블록을 읽어 RAM에 올린다. 이 시점에 커널은 "루트(/) = inode #2, inode 테이블은 블록 #8부터" 를 학습한다.

이후 `/` 로 시작하는 경로를 파싱할 때마다 커널은 inode #2를 시작점으로 삼는다. 경로를 탐색할 때마다 찾는 게 아니라 마운트 때 이미 알게 된 것이다.

루트 inode는 모든 절대경로 탐색에서 반드시 거치기 때문에 거의 항상 RAM의 inode 캐시에 올라와 있다. 루트를 통과할 때마다 디스크를 읽지 않는다.

### 단계별 흐름

`open("/home/alice/hello.txt")` 호출 시 OS가 거치는 단계를 단계별로 추적해보자.

## 간접 블록 포인터

### inode 크기 제약

inode는 크기가 고정이다. 보통 128바이트다. 메타데이터(크기, 권한, 타임스탬프, 소유자)가 이미 자리를 차지하면 데이터 블록 포인터에 쓸 수 있는 공간은 약 60바이트다. 포인터 하나가 4바이트면 직접 포인터 15개가 한계다.

```
15개 × 4KB = 60KB
```

60KB가 넘는 파일은 저장할 수 없다. 해결책은 포인터를 저장하는 블록 자체를 디스크에 두는 것이다.

### 직접 / 간접 포인터 계층

inode는 12개의 직접 포인터와 3개의 간접 포인터 슬롯을 갖는다.

```
inode (128바이트, 고정)
├── 직접 포인터 ×12   →  데이터 블록 (파일 내용)
├── 단일 간접 포인터  →  [포인터 블록]  →  데이터 블록 ×1024
├── 이중 간접 포인터  →  [포인터 블록]  →  [포인터 블록 ×1024]  →  데이터 블록 ×1024²
└── 삼중 간접 포인터  →  3단계 타고 내려감
```

간접 포인터 슬롯은 inode 안에 있지만, 포인터 블록(1024개의 포인터가 담긴 4KB 배열)은 디스크에 있다. 포인터 블록은 파일 데이터가 아니라 다른 블록의 번호들을 저장하는 용도의 블록이다.

inode 슬롯 1개(4바이트)로 4KB짜리 포인터 블록을 가리키면, 그 블록 안에 1024개의 포인터가 들어간다. 슬롯 1개가 1024배로 확장되는 구조다. 다단계 페이지 테이블이 CR3 레지스터 하나로 전체 가상 주소 공간을 커버하는 원리와 동일하다.

| 포인터 종류 | 커버 범위 | 데이터 블록까지 접근 횟수 |
|------------|----------|----------------------|
| 직접 ×12 | 48KB | 1번 |
| 단일 간접 | 4MB | 2번 |
| 이중 간접 | 4GB | 3번 |
| 삼중 간접 | 4TB | 4번 |

작은 파일이 절대 다수를 차지하기 때문에 직접 포인터 12개로 빠른 접근을 확보하고, 큰 파일은 추가 접근을 감수하는 구조다.

## Hard link와 Soft link

### 이름과 inode의 분리

inode에 이름이 없고 디렉토리가 이름을 들고 있다는 구조는 하나의 가능성을 만든다. 디렉토리에 같은 inode 번호를 가리키는 항목을 여러 개 추가할 수 있다.

```
디렉토리:
"hello.txt"  →  inode #42
"backup.txt" →  inode #42   ← 같은 inode
```

이것이 Hard link다.

### Hard link

Hard link는 기존 inode를 가리키는 디렉토리 항목을 추가하는 것이다. 파일 복사가 아니다. 데이터 블록을 새로 쓰지 않는다. 디렉토리 테이블에 줄 하나를 추가할 뿐이다.

inode는 자신을 가리키는 이름이 몇 개인지 참조 카운트(reference count)로 관리한다. 참조 카운트가 0이 될 때만 실제 데이터 블록이 해제된다.

```
파일 생성:         inode #42  참조카운트: 1  (hello.txt만 존재)
하드링크 생성:     inode #42  참조카운트: 2  (hello.txt, backup.txt)
hello.txt 삭제:    inode #42  참조카운트: 1  (backup.txt가 유지, 데이터 살아있음)
backup.txt 삭제:   inode #42  참조카운트: 0  (데이터 블록 해제)
```

원본과 링크의 구분이 없다. 둘 다 inode #42를 가리키는 동등한 이름이다. 어느 쪽을 통해 수정해도 같은 데이터 블록을 본다.

한계도 있다. 같은 파일 시스템 내에서만 사용 가능하다. inode 번호는 파일 시스템마다 독립적으로 관리되기 때문이다.

### Soft link (Symbolic link)

Soft link는 다른 원리로 동작한다. 새로운 inode와 데이터 블록이 만들어지고, 그 데이터 블록에 대상 파일의 경로 문자열이 저장된다.

```
"shortcut.txt"  →  inode #99  →  "/home/alice/hello.txt" (경로 문자열)
                                          ↓ (OS가 경로를 따라감)
                                  inode #42  →  [데이터 블록]
```

파일을 열 때 OS가 Soft link임을 감지하면 경로 문자열을 읽어 다시 이동한다. 원본 파일이 삭제되면 경로가 존재하지 않아 Soft link는 무효가 된다(dangling symlink). 반면 다른 파일 시스템이나 다른 서버의 경로도 가리킬 수 있어 Hard link보다 제약이 적다.

## 디스크 스케줄링

### HDD의 물리적 구조

SSD는 전기 신호만으로 임의 위치에 접근하지만, HDD는 헤드(read/write head)가 물리적으로 이동해야 한다. 원하는 트랙 위로 헤드를 옮기는 시간을 탐색 시간(seek time)이라 한다.

동시에 여러 I/O 요청이 들어오면 처리 순서에 따라 총 헤드 이동 거리가 크게 달라진다.

```
트랙:  0   10   20   30   40   50
헤드:              ↑ (현재 20번)
요청: 40, 5, 35, 15, 50
```

### 4가지 알고리즘

FCFS는 요청이 들어온 순서대로 처리한다. 구현이 단순하고 공평하지만 헤드가 앞뒤로 오가며 이동 거리가 길어진다.

SSTF는 현재 헤드 위치에서 가장 가까운 요청을 먼저 처리한다. 이동 거리는 짧지만 기아 현상이 발생한다. 가까운 요청이 계속 들어오면 먼 트랙은 무기한 대기한다. CPU 스케줄링의 SJF와 같은 구조다.

SCAN은 한 방향으로 끝까지 이동하면서 요청을 처리하고, 끝에 닿으면 방향을 바꾼다. 엘리베이터 알고리즘이라고도 부른다. 기아가 발생하지 않고 이동 거리도 효율적이다.

C-SCAN은 한 방향으로만 이동하고, 끝에 닿으면 처음 위치로 돌아가 다시 같은 방향으로 이동한다. 총 이동 거리는 SCAN보다 길지만 모든 트랙의 최대 대기 시간이 균일해진다. SCAN은 중간 트랙을 왕복 경로에서 두 번 서비스하는 반면 끝 트랙은 한 번뿐이다. C-SCAN은 이 불균형이 없다.

## 저널링

파일을 쓰는 도중 전원이 꺼지면 어떻게 될까.

inode의 블록 포인터를 업데이트한 직후 실제 데이터 블록에 쓰기 전에 꺼지면, inode는 블록이 있다고 기록하는데 실제 데이터는 없는 불일치 상태가 된다. 파일 시스템의 일관성이 깨진다.

저널링(Journaling)은 이 문제를 해결한다.

```
1. 저널(로그 영역)에 "이런 작업을 할 것이다" 먼저 기록
2. 실제 작업 수행
3. 저널에 "완료" 표시
```

재부팅 시 저널을 확인한다. "완료" 표시가 없으면 중간에 실패한 것이므로 작업을 다시 실행하거나 롤백한다. 데이터베이스 트랜잭션의 WAL(Write-Ahead Logging)과 같은 원리다.

Linux의 ext4, Windows의 NTFS 모두 저널링을 사용한다. fsck(File System Check) 같은 도구가 부팅 시 파일 시스템 일관성을 검사하는데, 저널링이 있으면 전체 디스크를 검사하지 않고 저널만 확인하면 돼서 복구 시간이 비교할 수 없이 짧다.
